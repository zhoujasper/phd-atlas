# Publishing the Team edition later

The public repository is deliberately shipped as a personal/self-hosted edition.
The private source repository remains the source of truth for the Team and
institution implementation while that surface is being prepared for public use.

Do not copy individual Team files into the public repository by hand. That can
leave navigation, API authorization, migrations, locale bundles, and tests out
of sync. Publish Team as one reviewed edition change through the private source
exporter.

## Readiness gate

Before enabling Team publicly, complete all of the following in the private
source repository:

1. Finish owner, teacher, student, invitation, reassignment, removal, audit
   recovery, and quota behavior.
2. Pass permission tests for every Team API route, including cross-team and
   unauthenticated requests.
3. Test migration of an existing personal SQLite database into a Team-capable
   release and test rollback to the prior release.
4. Test the complete Team surface at 320, 400, 820, and 1400 px, with keyboard
   navigation, reduced motion, dark mode, empty teams, and large teams.
5. Update the public README, data model notes, deployment guide, and release
   notes so administrators understand the new roles and backup implications.

## Source changes

Work in `zhoujasper/phd-atlas-source`, never directly in the generated public
mirror.

1. Create a branch such as `public-team`.
2. Change the public edition boundary from a Boolean to an explicit feature
   configuration, for example `TEAM_ENABLED=true`.
3. Remove Team files, styles, tests, and locale files from the exclusion list in
   `tools/export-public.mjs`.
4. Remove the compatibility stubs generated by `writePublicEditionFiles()`.
5. Enable Team navigation, invite routes, plan presentation, workspace
   bootstrap fields, and Team API routes together. Do not expose only the UI or
   only the API.
6. Add a database migration version and a downgrade/rollback test. Never make a
   public release depend on private demo-team seed data.
7. Export into a fresh directory and verify that no private files
   (`AGENTS.md`, `PROJECT_MEMORY.md`, `.public/`, or the private `docs/` tree)
   are present.

## Validation and publication

Run:

```bash
npm ci
npm run lint
npm run i18n:check
npx tsc --noEmit
npm test
npm run build
node tools/export-public.mjs ../phd-atlas-public-check
cd ../phd-atlas-public-check
npm ci
npm run i18n:check
npx tsc --noEmit
npm test
npm run build
```

After review, merge the source branch into `main`. The private repository's
`Sync public edition` workflow exports, validates, and pushes the generated
public tree. Do not merge a hand-edited public pull request over generated
files.

For a release:

```bash
git tag v1.1.0
git push origin v1.1.0
```

The tag is mirrored to the public repository. Its release workflow builds the
same update-package format accepted by **Admin → System information → System
update**, validates the package contents and rollback path, then attaches the
`.tar.gz` file and SHA-256 checksum to the GitHub Release.

## Emergency rollback

If the Team release fails its production smoke check, stop publication, fix the
private source, and issue a newer patch release. Existing native installations
can upload the previous compatible release package; the updater also retains a
local runtime rollback under `storage/update-rollbacks/`. Always retain a
separate whole-workspace backup before schema-changing releases.
